# 1 绪论

## 1.1 研究背景

随着人们生活水平的提高和饮食消费的升级，卤味食品作为一种具有浓厚地方特色的传统美食，越来越受到消费者的青睐。无论是休闲聚会、下酒佐餐，还是日常零食，卤味都占据着重要的地位。然而，传统的卤味门店经营面临着诸多挑战：手工记账效率低下、会员管理混乱、促销活动难以追踪、库存管理依赖经验判断导致浪费等问题日益突出。

在互联网+的时代背景下，传统餐饮行业向数字化转型已成为必然趋势。通过构建一套完整的线上管理系统，卤味门店可以实现从商品管理、订单处理、支付结算，到会员积分、优惠券营销的全流程数字化管理，从而降低运营成本、提升顾客体验、增强市场竞争力。

## 1.2 研究目的和意义

本研究旨在设计并实现一套面向香辣卤味门店的综合管理系统。系统涵盖顾客端（商品浏览、购物车、下单支付）和管理端（商品管理、分类管理、优惠券管理）两大部分，同时引入会员积分制度，通过积分奖励和等级晋升机制提升顾客粘性和复购率。

本系统的意义在于：
（1）规范门店运营流程，减少人工操作的错误率和时间成本；
（2）通过会员积分和优惠券体系，实现精准营销，提高顾客忠诚度；
（3）为门店管理者提供数据化的经营决策支持；
（4）探索传统餐饮行业数字化转型的技术路径，具有一定的推广价值。

## 1.3 平台选择与技术路线

本系统采用B/S（浏览器/服务器）架构，前端使用React + TypeScript + Ant Design构建响应式用户界面，后端基于Java Spring Boot 3.2框架提供RESTful API服务，数据库采用Microsoft SQL Server 2022保证事务一致性和数据安全性。前后端通过HTTP/JSON进行数据交互，采用JWT（JSON Web Token）实现无状态身份认证，MyBatis-Plus作为ORM框架连接数据库。

# 2 可行性分析

## 2.1 技术可行性

本系统采用的技术栈均为当前业界主流且成熟的方案。Spring Boot作为Java生态中最流行的微服务框架，具备完善的生态支持和社区活跃度；React作为前端主流框架，组件化开发模式能够大幅提升界面开发效率；SQL Server作为企业级关系型数据库，提供完整的事务管理、行级锁定和快照隔离机制，完全满足卤味门店订单处理的高并发和一致性要求。开发工具方面，使用Visual Studio Code作为代码编辑器，Maven进行项目构建管理，Git进行版本控制，技术方案成熟可靠。

## 2.2 经济可行性

系统的开发和部署成本较低。后端使用开源的Spring Boot框架，前端使用React和Ant Design，均为免费开源技术；数据库可使用SQL Server Express免费版本满足中小型门店的需求；部署服务器可采用门店内普通PC或云服务器（如阿里云、腾讯云入门级ECS）。系统上线后可显著提升门店运营效率，减少人力成本，通过会员营销提升营业额，投资回报率可观。

## 2.3 操作可行性

系统前端采用Ant Design组件库，界面简洁直观，符合用户操作习惯；前端做了移动端响应式适配，店长和店员可通过手机浏览器随时查看订单和管理商品；顾客端操作流程清晰（浏览→加购→下单→支付），学习成本低，无需专门培训即可上手使用。

# 3 需求分析

## 3.1 功能需求

本系统面向三类用户角色：顾客、店员（staff）和管理员（admin）。

**顾客端功能：**
- 浏览商品列表，按分类筛选，查看商品详情（含辣度、克数）
- 将商品加入购物车，管理购物车数量
- 下单结算，选择优惠券和积分抵扣
- 模拟支付，查看历史订单
- 查看积分余额和会员等级进度
- 领取优惠券，查看已持有的优惠券
- 对已完成的订单发表评价

**店员端功能：**
- 商品管理：新增、编辑、删除商品
- 分类管理：查看和调整分类
- 优惠券管理：查看优惠券领取和使用情况

**管理员端功能：**
- 拥有店员全部权限
- 可删除商品和分类
- 管理优惠券模板和发放策略

## 3.2 性能需求

（1）响应时间：普通页面加载时间应小于3秒，API接口响应时间应小于500毫秒；
（2）并发能力：系统应支持至少20个用户同时在线操作；
（3）数据处理：订单提交支持事务处理，确保在高并发场景下数据一致性；
（4）资源消耗：服务器CPU使用率不超过80%，内存使用率不超过70%。

## 3.3 可靠性和可用性需求

（1）系统应保证7×24小时运行，年可用性达到99%以上；
（2）数据库采用完整恢复模式，支持备份和还原操作；
（3）系统应具备乐观锁机制，防止并发操作导致的数据不一致；
（4）关键业务操作应有事务保障，异常时自动回滚。

## 3.4 出错处理需求

（1）用户登录失败：提示用户名或密码错误，明确告知错误原因；
（2）各页面数据加载失败：提示用户网络异常或服务器错误；
（3）下单失败：明确提示失败原因（库存不足/优惠券不可用/积分不足）；
（4）支付失败：提示支付状态异常，防止重复支付；
（5）Token过期：前端自动捕获401状态码，清除登录状态并跳转登录页。

## 3.5 接口需求

系统前后端通过RESTful API进行数据交互，接口规范如下：
- 协议：HTTP/HTTPS
- 数据格式：JSON
- 认证方式：JWT Bearer Token（Header: Authorization: Bearer xxx）
- 统一响应格式：{code: int, message: string, data: object}
- 开发环境前端端口5173，后端端口8080，通过Vite代理解决跨域

# 4 总体设计

## 4.1 基本设计概念和处理流程

### 4.1.1 基本设计概念

本系统采用前后端分离的B/S三层架构：表现层（React前端）、业务逻辑层（Spring Boot后端）、数据访问层（MyBatis-Plus + SQL Server）。表现层负责界面展示和用户交互，通过HTTP向业务逻辑层发送请求；业务逻辑层负责处理业务规则、权限校验、事务管理；数据访问层负责与数据库的CRUD操作。

系统采用JWT（JSON Web Token）实现无状态认证，用户登录后获得Token，后续请求携带Token进行身份验证。权限管理基于Spring Security的RBAC（基于角色的访问控制）模型，分为admin、staff、customer三种角色。

（插图：系统架构图.svg）

### 4.1.3 订单处理流程

订单处理是本系统的核心业务逻辑，采用8步事务保障数据一致性：
1. 库存校验与行级锁定扣减
2. 优惠券校验与原子锁定
3. 积分校验与扣减
4. 计算实付金额
5. 生成订单号
6. 创建订单与明细记录
7. 创建支付记录（pending状态）
8. 赠送积分并检查会员等级升级

以上8步在同一个数据库事务中执行，任一环节失败则全部回滚，同时配合@Retryable注解实现乐观锁冲突重试（最多3次，200ms退避）。

### 4.5.3 物理设计

系统的物理数据模型（PDM）基于SQL Server 2022，采用PowerDesigner逆向工程自动生成。所有金额字段使用DECIMAL(10,2)精确存储，时间字段使用DATETIME2(0)秒级精度，各业务表包含ROWVERSION字段实现乐观锁，is_deleted字段实现软删除，关键外键均建立索引。数据库文件配置为完整恢复模式，启用READ_COMMITTED_SNAPSHOT快照隔离级别减少读写阻塞。

## 4.6 系统出错处理设计

### 4.6.1 出错信息

（1）用户登录失败：提示用户名或密码错误，前端拦截空输入；
（2）数据加载失败：提示网络异常或服务器错误；
（3）下单失败：提示具体原因（库存不足/优惠券不可用/积分不足）；
（4）Token过期：401状态码，自动清除本地存储，跳转登录页；
（5）权限不足：403状态码，页面显示无权访问。

### 4.6.2 补救措施

（1）后备：数据库每日自动完整备份，保留30天；差异备份每6小时一次；日志备份每15分钟一次；
（2）恢复：使用RESTORE DATABASE命令从完整备份+差异备份+日志备份链恢复到最近时间点；
（3）事务保障：下单操作全程在一个数据库事务中执行，提交前任一环节失败自动回滚。

# 5 详细设计

## 5.1 程序的系统结构

程序从底至上分为：数据持久层（SQL Server）、ORM层（MyBatis-Plus）、业务服务层（Spring Service）、控制层（Spring Controller）、前后端HTTP接口层、前端展示层（React组件）。各层之间通过接口解耦，Service层处理业务逻辑和事务，Controller层封装RESTful API，前端通过Axios调用后端接口。

（插图：系统架构图.svg）

## 5.2 用户认证模块设计

### 5.2.1 需求描述

用户认证模块负责用户的注册、登录、身份验证和权限控制。系统使用JWT无状态认证，用户成功登录后获得签名的JWT令牌，后续访问携带令牌即可完成身份验证。

### 5.2.2 流程图

用户认证模块的流程图展示了从访问系统到完成认证的完整流程，包括注册分支（新用户填写信息→BCrypt加密存储→返回令牌）、登录分支（输入凭证→验证哈希→生成JWT）以及权限拦截机制。

（插图：用户认证模块.svg）

### 5.2.3 接口设计

| 描述 | 用户注册 | 用户登录 |
|------|----------|----------|
| 地址 | POST /api/auth/register | POST /api/auth/login |
| 入参 | username,password(≥6位),email | username,password |
| 出参 | {token,username,role,userId,membershipLevel} | 同左 |
| 认证 | 无需认证 | 无需认证 |

### 5.2.4 处理逻辑

（1）注册时先查重用户名，不存在则创建用户，密码通过BCryptPasswordEncoder编码存储；
（2）登录时查用户表，使用matches()方法验证密码哈希；
（3）登录成功后生成JWT令牌，payload包含userId、username、role、membershipLevel，有效期24小时；
（4）前端将令牌存入localStorage，Axios拦截器自动在请求头添加Authorization: Bearer xxx。

## 5.4 订单与支付模块设计

### 5.4.1 需求描述

订单模块是系统最核心的业务模块，涉及库存扣减、优惠券核销、积分抵扣、订单创建和支付确认等复杂操作。系统采用数据库事务保障这8步操作的原子性，并配合乐观锁重试机制应对并发场景。

### 5.4.2 流程图

### 5.4.4 处理逻辑

下单事务使用@Transactional注解保证原子性，@Retryable注解实现乐观锁冲突重试。库存扣减使用UPDATE ... WHERE stock >= qty的原子SQL，优惠券锁定使用UPDATE ... WHERE status='unused'，积分扣减后重新计算会员等级。支付模块执行校验（订单归属、支付状态）后更新支付状态和订单状态。

## 5.5 积分与会员模块设计

### 5.5.1 需求描述

系统建立了铜牌→银牌→金牌→铂金四级会员体系。用户消费后赠送积分（消费额的1%），积分累积达到阈值自动升级。不同等级享受不同的积分抵扣汇率（铜100:1、银90:1、金80:1、铂70:1），等级越高积分越值钱。此外，高等级会员可领取专属优惠券。

### 6.7.2 控制流分析

方法的主要判定节点：

- D1：库存是否充足 → 不足则抛异常回滚
- D2：是否使用优惠券 → 跳过或校验
- D3：优惠券是否可用 → 不可用则抛异常回滚
- D4：是否使用积分 → 跳过或校验
- D5：积分是否充足 → 不足则抛异常回滚
- D6：会员等级是否提升 → 写升级记录或跳过

# 7 总结与展望

本项目基于Spring Boot + React + SQL Server技术栈，成功开发了一套面向香辣卤味门店的综合管理系统。系统实现了用户认证、商品管理、订单处理、支付模拟、积分与会员管理等核心功能，采用JWT认证、数据库事务、乐观锁、行级锁等技术保障了系统的安全性和数据一致性。

**本项目的主要成果和创新点包括：**

（1）完整的交易闭环：从商品浏览→购物车→下单→支付→订单跟踪，一套完整的O2O交易流程；
（2）四级会员制度：铜牌→银牌→金牌→铂金的分级会员体系，积分汇率差异化设计（等级越高积分越值钱），配合自动升级和升级提醒机制，有效激励顾客复购；
（3）并发安全保障：库存扣减使用行级锁+乐观锁双重机制，优惠券使用原子SQL的CAS操作，防止超卖和券的重复使用；
（4）前后端分离架构：React前端配合Spring Boot后端，Vite代理解决跨域，Redux管理状态，组件化开发提高可维护性；
（5）完整的软工文档：包含CDM、PDM、E-R图、系统架构图、功能结构图、数据流图、模块流程图等全套设计文档。

**本系统存在以下不足和展望：**

（1）支付为模拟支付，后续可接入真实支付接口（如微信支付、支付宝）；
（2）未实现文件上传功能，商品图片目前使用静态资源；
（3）可增加数据报表和可视化看板功能，为管理者提供销量分析、顾客画像等数据支持；
（4）可扩展微信小程序端和移动APP端，形成多端覆盖；
（5）可引入消息推送功能，向顾客推送优惠券到期提醒、积分变动通知等。
